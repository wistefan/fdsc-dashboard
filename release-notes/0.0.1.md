# Release 0.0.1

Label-driven semantic versioning — every PR must carry exactly one of major, minor, or patch; compute-next-version.sh derives the next vMAJOR.MINOR.PATCH from the latest git tag plus the chosen bump label.
PR validation workflow (.github/workflows/pr-labels.yml) — enforces the version label and a PR-description format (## Summary, ## Release Notes sections) via check-labels.sh and check-pr-description.sh.
Test workflow (.github/workflows/test.yml) — runs on every push and PR: npm ci, npm run lint, npm run build, and npm test --if-present.
PR build workflow (.github/workflows/pr-build.yml) — builds and pushes a multi-arch (linux/amd64, linux/arm64) image to quay.io/seamware/fdsc-dashboard tagged <nextVersion>-PRE-<shortSha>.
Release workflow (.github/workflows/release.yml) — on push to main, computes the new version, tags the commit, pushes the multi-arch image with the version tag and latest, writes release-notes/<version>.md (from a dedicated release-notes/next.md file if present, otherwise from the PR ## Release Notes section), regenerates the RELEASE-NOTES.md index table, and creates a GitHub Release.
PR template (.github/pull_request_template.md) — enforces the description format.
Dedicated release-notes override — contributors can ship release-notes/next.md on the PR branch for richer notes; the release workflow renames it to release-notes/<version>.md on merge.
Release-notes index (RELEASE-NOTES.md) — auto-generated Markdown table of version → date → note link.
